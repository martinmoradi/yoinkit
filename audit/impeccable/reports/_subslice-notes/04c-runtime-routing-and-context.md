## Paradigms Worth Importing

- **Router skill, but with honest lazy-load wording.** YoinkIt can steal the one `/yoink` router and lazy `reference/map.md` / `reference/capture.md` structure, but the always-loaded manifest should say which references are mandatory together: command flow plus capture law plus any current project/driver context.
- **Explicit directive protocol.** The `NO_PRODUCT_MD` and `NEXT STEP` stdout strings are stronger than absence or implicit state. A YoinkIt boot script should print named next actions like `NO_VISIBLE_BROWSER`, `NEXT STEP: run map`, or `CAPTURE_BLOCKED: open a headed browser`, instead of expecting the agent to infer from empty JSON.
- **Cheap signals, no capture.** `context-signals.mjs` is a good model for a `yoink-signals.mjs`: gather libraries, candidate selectors, viewport, prior manifest, and whether a visible capture browser/session exists; do not arm or capture inside the gatherer.
- **Trace assertions over prose assertions.** The skill-behavior harness is directly transferable. YoinkIt should assert that the agent calls `on`/`scan` before the real trigger, waits long enough, and calls `dump` after the wait. That catches the measured "armed mid-transition, captured nothing" failure better than checking the final explanation.
- **Update check as boot piggyback.** The update channel is nicely bounded: opt-out, once/day poll, once/week re-notify, 1.2s timeout, silent failure, user consent before running `npx impeccable update`. If YoinkIt gains a once-per-session script, this rides for free.
