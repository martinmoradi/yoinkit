## Executive Delta

- Verified against freshly cloned upstream `pbakaus/impeccable` at commit `1c897a0`. The report is broadly accurate, the cited source files exist, and the important line anchors still point at the claimed symbols.
- The main correction is precision, not reversal: Tier 2 is an in-page visual resolver that uses canvas for drawable underlays, but gradients are handled by parsed-stop worst-contrast math and solid colors by computed color parsing. Do not describe all Tier 2 results as "actual painted pixels."
- The failure-mode matrix should split semi-transparent background-color layers from CSS `opacity` stacks. The former can be recursively blended in Tier 2; the latter is an explicit screenshot-only bailout.
- The color parser breadth is slightly overstated. `isNeutralColor` can classify several modern color functions, but RGB conversion for contrast currently covers `rgb/rgba`, hex, and `oklch`; the design-system parser separately adds `hsl`.

## Deep Implementation Notes

- Confirmed `isNeutralColor` is exactly the fail-open lesson the report says it is. It treats missing/transparent as neutral, checks RGB channel spread `< 30`, OKLCH chroma `< 0.02`, LCH chroma `< 3`, OKLab/Lab `hypot(a,b)` thresholds, HSL saturation `< 10`, and HWB chroma-like saturation `< 0.1`; unrecognized formats return `false` so they are detected rather than skipped (`cli/engine/shared/color.mjs:3-52`).
- Confirmed WCAG math is shared conceptually: `relativeLuminance` and `contrastRatio` implement standard sRGB linearization and `(max + 0.05) / (min + 0.05)` (`shared/color.mjs:61-72`), while screenshot diff reimplements the same math inside `page.evaluate` (`cli/engine/engines/visual/screenshot-contrast.mjs:42-53`).
- Confirmed the static color-resolution chain: `buildCustomPropMap(document)` walks accessible stylesheets, descends into `@media`/`@supports`, and collects custom props from `:root`, `html`, or `:host` rules (`cli/engine/rules/checks.mjs:866-903`). `resolveVarRefs` substitutes chained `var(--x[, fallback])` up to depth 8 (`checks.mjs:905-917`). `parseColorResolved` runs var resolution then `parseAnyColor` (`checks.mjs:995-1002`).
- Confirmed OKLCH conversion and the Tailwind v4 rationale. `oklchToRgb` uses the OKLab intermediary matrices, gamma encodes, clamps, and returns `{r,g,b,a:1}` (`checks.mjs:919-946`). The comment explicitly says Tailwind v4's OKLCH palette would be invisible to contrast/color checks without this conversion (`checks.mjs:921-924`).
- Confirmed Tier 1 `checkColors` gates: styled-button exception for `a`/`button` with own opaque background and direct text (`checks.mjs:65-78`), worst contrast ratio across gradient stops (`checks.mjs:81-100`), static-only alpha fallback skip (`checks.mjs:101-115`), emoji-only skip via `isEmojiOnlyText` (`checks.mjs:53-63`), and gray-on-color only when every considered background stop is chromatic (`checks.mjs:86-91`).
- Confirmed Tier 2 candidate selection: `collectVisualContrastReasons` records `background-clip text`, `text shadow`, image/gradient backgrounds, CSS `opacity`, blend/filter/backdrop-filter, and image/canvas/video/svg underlays discovered with `elementsFromPoint` (`cli/engine/browser/injected/index.mjs:600-653`). `collectVisualContrastCandidates` requires rendered direct text, skips safe inline tags unless styled as buttons, computes selector/threshold/clip/textColor, and marks `preferRenderedForeground` for cases where declared color may be unreliable (`injected/index.mjs:655-713`).
- Confirmed Tier 2 sampling detail: drawable rasters are cached in a `WeakMap`, capped to 640px on the long side, and taint/cross-origin errors become unresolved reasons rather than thrown failures (`injected/index.mjs:717,900-953`). `resolvePaintedImageRect` and `resolveObjectImageRect` map viewport points through `background-size`/`background-position` and `object-fit`/`object-position` (`injected/index.mjs:784-854`). `sampleVisualBackgroundAtPoint` walks the `elementsFromPoint` stack and can recursively blend semi-transparent sampled background colors over underlayers, depth-capped above 8 (`injected/index.mjs:1026-1074`).
- Confirmed Tier 2 verdict behavior: it bails to screenshots before sampling for `background-clip text`, blend/filter/backdrop-filter, CSS `opacity stack`, and text shadow (`injected/index.mjs:1097-1106`). It samples an inset grid of text points, requires at least `min(3, points.length)` readable samples, reports p10 plus median, and emits `low-contrast` only for measured failures (`injected/index.mjs:1008-1024,1123-1164`).
- Confirmed lazy/offscreen behavior: `analyzeVisualContrast` can scroll offscreen candidates into view and restore scroll when `scrollOffscreen` is true (`injected/index.mjs:1173-1203`). Separately, `scheduleLazyVisualContrast` watches `text outside viewport` unresolved candidates with `IntersectionObserver` only when lazy mode is allowed and `scrollOffscreen === false`; it guards stale work with `scanGeneration` and updates decorated/extension findings when resolved (`injected/index.mjs:1683-1740`).
- Confirmed Tier 3 screenshot diff: `sanitizeScreenshotClip` clamps to viewport width and caps height at 320px (`cli/engine/engines/visual/screenshot-contrast.mjs:1-15`). `captureVisualContrastCandidate` screenshots with `captureBeyondViewport: true`, marks only the target selector with `data-impeccable-visual-contrast-target`, hides text via transparent color and text-fill, removes background image for background-clip text, captures again, and cleans attributes in `finally` (`screenshot-contrast.mjs:108-170`).
- Confirmed screenshot metrics: changed pixels with channel delta `< 10` are ignored, changed pixels are treated as glyph pixels, foreground comes from declared text color unless `preferRenderedForeground` is set, background comes from the after screenshot, ratios are sorted, and p10/median are returned only when at least 8 ratios exist (`screenshot-contrast.mjs:55-104`). `captureVisualContrastCandidate` returns no finding when p10 is missing, glyph pixels `< 8`, or p10 passes threshold (`screenshot-contrast.mjs:173-182`).
- Confirmed orchestration: `runVisualContrastFallback` first calls `window.impeccableAnalyzeVisualContrast`, collects its fail findings unless a selector already has normal `low-contrast`, then filters out selectors already resolved as pass/fail before running `captureVisualContrastCandidate` on the remaining unresolved candidates (`cli/engine/engines/browser/detect-url.mjs:29-99`). This supports the report's "clean handoff" framing.

## Paradigms Worth Importing

- Keep the fail-open default. `isNeutralColor` is the clearest one-line transfer to YoinkIt: unknown or unparsed motion/color state should become visible capture or verification work, not a silent skip.
- Copy the named-handoff pattern more than the exact contrast code. Each unsure tier returns an explicit reason such as `stale selector`, `text outside viewport`, `tainted image`, or `<effect> needs screenshot pixels`; YoinkIt can use the same shape for `measured` versus `verify` capture confidence.
- Treat render-twice/diff as a verification layer. Impeccable isolates glyph pixels by hiding target text; YoinkIt could isolate animation-touched pixels by diffing adjacent frames or baseline/triggered states, especially for canvas, filters, blend modes, and compositor effects that computed style underspecifies.
- Reuse the source-pixel geometry only where it matches YoinkIt's needs. `resolvePaintedImageRect` and `resolveObjectImageRect` are good references for "which source pixel is under this viewport point," but the report should not imply this gives painted gradient pixels.
