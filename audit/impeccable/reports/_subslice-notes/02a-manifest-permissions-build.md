## Paradigms Worth Importing

- Keep the "derive, do not fork" pattern for future YoinkIt cross-browser work. A Chrome manifest can stay canonical while a build step derives the Firefox event-page form and AMO metadata. The service worker must stay compatible with that transform.
- Adopt the generated metadata sidecar pattern when YoinkIt has capture modes, trigger presets, or rule metadata that the panel needs. Generate the panel data from the engine source, then have the UI fetch it rather than duplicating lists by hand.
- If YoinkIt keeps generated artifacts ignored, use release guards that inspect the actual built artifact or run a smoke test, not `git status` on ignored paths. If a generated browser bundle is package-facing and tracked, use the Impeccable CI pattern: rebuild, then `git diff --exit-code` on the tracked bundle.
- Preserve the frugal extension posture, with one nuance. `activeTab` plus `optional_host_permissions` is a good install-prompt hardening path for YoinkIt, but it would need an explicit `chrome.permissions.request` flow before all-origin capture. `activeTab` alone only grants temporary access to the current active tab after user gesture.
- The single-source engine idea is transferable, but YoinkIt can avoid Impeccable's duplicate `browserSafeModule()` implementation by extracting a shared bundling helper from the start.
