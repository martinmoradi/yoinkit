## Executive Delta

- Confirmed the core 03b framing: live-mode durable state is an append-only per-session JSONL journal plus a derived snapshot fold. `appendEvent` appends synchronously before folding, `getSnapshot` rebuilds from the journal on every read, and `restorePendingEventsFromStore` re-enqueues journaled pending work on server boot.
- Confirmed the important correction that `applyEvent` is not a guarded FSM. It is an event-type dispatch that rewrites phase unconditionally, except for `checkpoint`, which is guarded by terminal phase and monotonic `revision`.
- The biggest integration correction is the manual-edit seam. `session-store.mjs` has a `manual_edit_apply` fold case and `live-resume.mjs` has a manual-apply hint, but the current runtime does not journal manual-apply chat-route events. They are enqueued directly in `state.pendingEvents` and their replies return before the generic session-store reply journaling path. The parent overview is also too absolute when it says manual edits do not use the poll-loop agent: source now supports a chat route that dispatches `manual_edit_apply` over `live-poll.mjs`.
- The report should not describe `agent_error` as terminal-hidden. The store's `COMPLETED_PHASES` contains only `completed` and `discarded`; `agent_error` clears pending work but still appears in `listActiveSessions()`.
- `live-complete.mjs` has a direct-journal fallback only when the server is unavailable or `/poll` fails. If the server returns OK after swallowing a reply journal failure, `live-complete.mjs` does not append a local fallback.

## Deep Implementation Notes

- `appendEvent` is accurately described as append-before-fold: it normalizes the event, copies a legacy journal forward if needed, loads the prior folded state, derives `seq` from `prior.nextSeq`, writes one JSONL line with `fs.appendFileSync`, folds with `applyEvent`, refreshes the cache, and rewrites the snapshot in `skill/scripts/live/session-store.mjs:33-54`.
- `getSnapshot` really is a fresh replay read, not a cache hit. It calls `rebuildSnapshotFromJournal`, refreshes the in-memory cache, rewrites the snapshot mirror, then hides only completed/discarded sessions unless `includeCompleted` is set in `skill/scripts/live/session-store.mjs:56-64`.
- `rebuildSnapshotFromJournal` keeps replay deterministic: it starts from `baseSnapshot`, tracks `nextSeq` from parsed entries, folds each entry, and records malformed lines as diagnostics instead of throwing in `skill/scripts/live/session-store.mjs:128-152`.
- `applyEvent` structurally copies snapshot fields and sets `updatedAt` from `entry.ts` before the switch in `skill/scripts/live/session-store.mjs:155-164`. The event dispatch table in the leaf matches source: `generate` sets `generate_requested`; `agent_done`/`variants_ready` branches on `carbonize`; `accept`, `steer`, and `discard` set pending work; `complete` and `discarded` clear pending work; default emits `unknown_event_type`.
- The checkpoint guard is exactly the leaf's strongest point: completed/discarded snapshots reject checkpoints, otherwise `revision >= checkpointRevision` can update phase, owner, visible variant, files, preview mode, and params; stale checkpoints append `stale_checkpoint_ignored` in `skill/scripts/live/session-store.mjs:196-213`.
- The inbound server path enforces durability as a precondition. `/events` parses JSON, checks token, rejects manual-edit event types, validates, then calls `state.sessionStore.appendEvent(msg)` before `enqueueEvent(msg)`, returning HTTP 500 on append failure in `skill/scripts/live-server.mjs:656-705`.
- The reply path is deliberately weaker: after `acknowledgePendingEvent`, it maps reply status to `steer_done`/`discarded`/`complete`/`agent_error`/`agent_done` and appends in a `try/catch` that does not fail the HTTP reply in `skill/scripts/live-server.mjs:879-924`.
- Recovery printers match the leaf with the manual-apply caveat above. `live-resume.mjs` selects the first active session or `--id`, prints inactive when none exists, then derives `nextAction` from `pendingEvent`, `carbonize_required`, `accept_requested`, or an inspect fallback in `skill/scripts/live-resume.mjs:70-88`. `live-status.mjs` returns `null` for server fetch failures and falls back to journal-derived `activeSessions` in `skill/scripts/live-status.mjs:14-44`.
- Browser-side advisory state is also accurately represented. `live-browser-session.js` creates `prefix-session`, `prefix-session-handled`, and `prefix-session-scroll`, wraps storage operations in `try/catch`, and maintains `checkpointRevision` in `skill/scripts/live-browser-session.js:11-123`. The overlay sends checkpoint payloads with `revision`, `owner`, phase, files, visible variant, and params in `skill/scripts/live-browser.js:6233-6249`, then `resumeSession()` restores from localStorage and queues a checkpoint on reload in `live-browser.js:7952-8053`.
- File counts in the 03b leaf file map match the canonical clone: `session-store.mjs` 289, `live-server.mjs` 1134, `live-resume.mjs` 94, `live-status.mjs` 61, `live-complete.mjs` 75, `live-browser-session.js` 123, and `impeccable-paths.mjs` 126 lines.

## Paradigms Worth Importing

- The YoinkIt transfer note is directionally right: append-before-act is the most portable idea. For capture, "a capture you cannot persist is not captured" should mean the spec write and journal append both succeed before a source x viewport pair moves into `donePairs`.
- The rebuild-on-read pattern is worth stealing because it is cross-process by construction. A future `yoinkit-resume` should derive its instruction from an on-disk fold, not from a coordinator process that may have died.
- The canonical journal/advisory browser memory split maps neatly onto YoinkIt's real-browser constraint. Browser-local state can make the overlay pleasant after reloads, but the source-of-truth coverage ledger should be the disk journal.
- If YoinkIt imports manual-apply-like chat dispatch, do not copy Impeccable's current non-durable in-memory pending-event route by accident. Either document it as volatile, or journal it before enqueueing so restart recovery actually applies.
- Preserve the leaf's warning against a fake guarded FSM. The useful implementation pattern is deterministic replay plus strict append ordering; transition validation belongs upstream unless YoinkIt deliberately chooses a guarded fold.
